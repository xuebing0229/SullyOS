# 生活三层 · 家园/日程/小屋统一设计

> **状态：设计稿（design-only）。本文只定方向、不含实现改动。** 目标：把目前三套各自为政的「角色生活」系统——日程 / 家园 / 小屋——收敛成一条 **上游约束下游** 的派生链，让角色有一致、连贯、不自相矛盾的社会与生活。
> 动这三块（`scheduleGenerator.ts` / `utils/worldHome/*` / `RoomApp.tsx`）任意一处前，先读本文对齐心智模型。

## 一句话

**家园定义「在哪·和谁」，日程定义「何时·做什么」，小屋是「此刻这个空间的定格」。** 信息只往一个方向流：家园 → 日程 → 小屋。下游从上游派生，所以结构上就不可能相悖。

---

## 背景：为什么现在感觉「被拆散」

三个功能各自是一个**独立的内容生成器**，各生成各的、各刷各的、各有各的时间模型，底层数据零共享（只在 `RoomApp` 的 Tab 上并排）：

| 功能 | 主文件 | 现状 |
|---|---|---|
| **日程** | `utils/scheduleGenerator.ts` | LLM 生成一天 5-7 个时段 slot + 早/中/晚意识流 `flowNarrative`，注入聊天上下文（`utils/context.ts:407 buildScheduleInjection`）。喂了记忆宫殿 + core context，但**没读家园**。 |
| **家园** | `apps/WorldHomeApp.tsx` + `utils/worldHome/*` | 独立多角色世界引擎，自己的 tick 调度（早/午/晚），自己的 `real`/`sim` 时间模型。`real` 模式经 `world_card` 写回各角色 1v1 聊天 + 记忆宫殿。**不使用 `DailySchedule`。** |
| **小屋** | `apps/RoomApp.tsx` | 每天 6:00（`getVirtualDay` 以 6 点分界）一次性大批量 LLM（`initializeRoomState`，`RoomApp.tsx:533`），把房间状态 + **每一件家具的描述和反应** + 今日 todo + 随笔全生成一遍。就是「扯淡的 long 加载」的来源。 |

> 注意：`apps/ScheduleApp.tsx` 是**任务/纪念日管理器**，与「角色一天在做什么」的日程系统无关，不要被名字误导。

问题不是「功能不够」，是**缺一条把它们串起来的主线**。三个生成器各自脑补「角色的生活」，自然容易互相打架（早上聊天说要上班，日程却安排在家刷手机，小屋又显示在健身房）。

---

## 核心模型：生活三层派生链

用做游戏的思路，这三个东西天然是同一件事的**三种分辨率**：

| 层 | 是什么 | 回答 | 变化速度 | 现有载体 |
|---|---|---|---|---|
| **家园** = 世界/社会层 | 你住在哪、和谁住、你的社会关系 | **在哪 · 和谁** | 慢（世界级） | `WorldProfile`（`types.ts:1164`） |
| **日程** = 时间层/一天的脊椎 | 今天从早到晚干嘛、此刻在做什么 | **何时 · 做什么** | 每天 | `DailySchedule`（`types.ts:508`） |
| **小屋** = 空间层/此刻的定格 | 此刻这个空间长啥样、角色在里面干嘛 | **此刻 · 这里** | 实时/增量 | `roomConfig` + `RoomGeneratedState`（`types.ts:1976 / 522`） |

**派生规则（下游不许跟上游相悖）：**

```
家园（主家园：住所 + 同住人 + 世界此刻时段）
   │ ① 慢事实喂进日程生成 prompt（住哪·和谁）        ┌─ ② 家园演绎时，当事人日程已在其 system prompt 里
   ▼                                                  │    （复用 buildChatRequestPayload 白送，见 §日程层②）
日程（今天的 slots；每个 slot 有 activity/location/innerThought）
   │ ③ 生成 B 时看齐已生成的同住人 A（后者向前者看齐）
   │ 小屋直接读当前 slot
   ▼
小屋（用户拼的空间画布 + 读日程当前 slot 渲染「此刻角色在做什么」+ 涓流 ambient delta）
```

主线是 **家园 → 日程 → 小屋** 的单向派生；② 和 ③ 是两条"回看/横看"的对齐轴，防止另外两个方向也打架（详见「日程层」三轴表）。

这条链给出的游戏感：**家园是社会，日程是这一天的传记，小屋是你趴窗户往里看的那一眼**——三者不打架，因为它们是一条派生链，不是三个独立脑补。

## 三层详细定义

### 1. 家园层：主家园唯一性

**痛点（代码现状）**：`engine.ts:275` 只判 `world.timeMode !== 'sim' && world.injectToChat !== false`。一个角色可以同时是**多个 `real` 世界**的成员，每个都往记忆里写 `world_card` —— 会灌进互相打架的「我住哪 / 和谁住」，日程和小屋跟着一起精分。

**设计**：引入**「主家园」（canonical / primary home）**概念——

- 一个角色**有且仅有一个**对齐真实时间、注入记忆的主家园。它是日程和小屋的**唯一上游事实源**。
- 角色参与的其余世界**必须是 `sim`**（虚拟时间、不进记忆、当番外/平行故事线），不喂日程、不进主记忆、不进小屋。
- **唯一性由用户手动指定**（本轮已定的方向）：在角色档案里加「主家园」选择器，用户明确钦点哪个 `real` 家园是 canonical。
  - 检测到一个角色同时挂在多个 `real` 世界时：**弹提示让用户选**，不自动降级——把决定权留给用户。
  - 未指定主家园时：日程/小屋退回「无家园」行为（即现状），不强塞。

> 数据落点（待实现时定）：候选是 `CharacterProfile.primaryHomeId?: string` 指向某个 `WorldProfile.id`；或在 `WorldProfile` 侧标 canonical 成员。倾向前者——「主家园」是角色的属性，不是世界的属性。

### 2. 日程层：三条「不打架」对齐轴

「别相悖」不是一条线，是**三条对齐轴**。三条轴的现状差别很大，**别一视同仁地全当成"要新做"**——其中一条已经白送了。

| 轴 | 谁看谁 | 防的是 | **真实现状** |
|---|---|---|---|
| ① 日程 ← 家园 | 生成日程时读主家园（住哪·和谁·时段） | 日程说的家跟家园不符 | ❌ **要做**。见下 ①。 |
| ② 家园演绎 ← 当事人日程 | 家园演角色 X 时看 X 的日程 | 家园演的内容跟日程硬撞 | ✅ **已白送**。见下 ②。 |
| ③ 日程 ← 同住人日程 | 生成 B 的日程时看已生成的同住人 A | 一家人当天各过各、互相打脸 | ❌ **要做**。见下 ③。 |

#### ① 日程 ← 家园（要做）

**现状**：`scheduleGenerator.ts` 生成日程时读了记忆宫殿 + `buildCoreContext`，但**没显式读角色的主家园**（住所、同住人、家园此刻时段）。只能靠家园写回记忆后、偶然被记忆宫殿召回的碎片去猜「我住哪」，不可靠。

**设计（最低成本、最高收益）**：把**主家园的结构化事实**明确塞进日程生成 prompt：

- 你住在 `{主家园.name}`，和 `{同住人}` 同住。
- 家园此刻处于 `{早/午/晚}`（对齐 `WorldTimeMode`）。
- 家园最近的世界 beat 摘要（**已拍板：喂**——每天生成时只喂一次，成本可控，连贯性优先）。

有了这几条，`buildLifestylePrompt` / `buildMindfulPrompt` 生成的 slot 会自然落在正确的居所和社会关系里，`location` 也和家园一致。

#### ② 家园演绎 ← 当事人日程（已白送，仅需微调）

**这条不用重做——现在就已经生效。** 关键事实：家园演每个角色时**复用了主聊天的打包函数** `buildChatRequestPayload`（`engine.ts:325 / 559`，注释明写「复用聊天主链路」），而**日程注入就长在这条链路里**（`chatPrompts.ts:209` 加载该角色当天日程 → `chatPrompts.ts:313` `buildScheduleInjection` 拼进 system prompt）。所以：

> 家园演角色 X 时，X 自己的当天日程（含「当前时段硬事实」）**早就在它的 system prompt 里了**。

「每个时段必看一眼」这个诉求，**基本已经满足**（`buildScheduleInjection` 每轮都注入当前 slot）。唯一要留意的小坑：家园若是 `sim`（虚拟时间）模式，注入的"当前时段"是按**真实时间**算的、可能对不上虚拟时间——但 `sim` 是番外、不进主日程，主家园（`real`）不受影响，**基本不用管**。

> ⚠️ 历史修正：本文早期草稿曾写「家园完全读不到日程」，**错误**。worldHome 目录里确实没有直接引用 `DailySchedule`，但它间接经 `buildChatRequestPayload` 拿到了。别再被"grep 不到"误导。

#### ③ 日程 ← 同住人日程（要做）

**现状**：`scheduleGenerator` 是**一个角色一个角色单独生成**的，彼此不看。于是同住的 A、B 当天可能对不上（A 日程写「19:00 和 B 一起做饭」，B 却写「19:00 独自健身」）。

**设计：同住角色的日程要互相咬合，规则是「后生成的向已生成的看齐」。**

- 生成谁的日程，就先把**主家园里同住人**中**今天已经生成好日程**的那几份捞出来，作为「他们今天的安排」喂进 prompt，要求尽量接得上。
- **锚是涌现的，不用配置**（已拍板）：**当天谁的日程先生成，谁就是锚**。日程本来就是懒生成（用户先打开谁的聊天谁先生成），等于「用户今天最先想见的角色定全家节奏」。实现上不需要任何排序逻辑，规则只有一条：**生成时看齐所有今天已生成的同住人**。就算今天只对齐一半，**次日互相参考、越滚越一致**，自然收敛。
- **范围圈死**：只跟**主家园里的同住人**咬合，不跟所有角色（否则爆炸），语义上也只有"住一起的人"才需要当天对得上。
- **喂精简摘要**：同住人日程别整份灌进去，挑关键 slot（尤其提到当事人或共同活动的），控制 prompt 体积。

> ⚠️ **读法别用错**：③ 要读同住人的**完整 `DailySchedule.slots`**（从 `DB.getDailySchedule` 捞原始整份），**不是** `buildScheduleInjection` 吐的那段。后者故意只露「当前时段 + 下一个 + 当前心境」（`context.ts:407-463`，为免角色像念课程表），看不到全天——而 ③ 要对齐的恰恰是全天（如 A 晚上和 B 的共同安排）。
> 记住两种用途：**「注入版日程」= 给聊天/家园演绎的"此刻快照"（当前 slot）；「原始日程」= 给 ③ 用的"整天蓝图"（全 slots）。**

### 3. 小屋层：去掉每日全刷，改「读日程 + 懒加载 + 涓流」

**痛点**：`initializeRoomState`（`RoomApp.tsx:533`）每天 6:00 大批量 LLM——房间状态 + **每件家具描述和反应** + todo + 随笔全生成一遍。物件越多越久，纯纯的长加载。家具是静态的，天天重生成描述毫无意义。

**设计（三管齐下）**：

1. **「角色此刻在做什么」直接读日程当前 slot。**
   小屋本就在读 `roomSchedule`（`RoomApp.tsx:431/443/552` 已有 `DB.getDailySchedule`）。当前 slot 的 `activity / location / innerThought` 就是「此刻角色在这空间里的状态」，**不需要小屋自己再生成一遍 `actorStatus`**。→ 砍掉 `initializeRoomState` 里生成 actor 状态的部分。

2. **家具描述改「点击时懒生成 + 永久缓存」。**
   静态家具的描述/反应不需要每天刷。改成：点哪件才生成哪件，写进 `RoomGeneratedState.items[id]` 后**永久缓存**（除非家具本身被换/删）。→ 干掉「一次性把所有家具全生成」的大批量调用，长加载消失。

3. **「生活刷新了」靠副 API 涓流，不靠长加载。**（见下一节）

**结果**：进小屋不再有大 loading；家具描述按需长出来并缓存；角色状态实时跟日程走；生活动态由 ambient delta 涓流补充。

## 副 API（情绪 API）涓流：ambient 生活动态

**观察**：情绪 eval（`emotionConfig`）本来就在每轮聊天后跑，空闲/关闭时整条短路（`useChatAI.ts:451` 双 gate `isScheduleFeatureOn(char) && emotionConfig?.enabled`）。它有富余产能。

**设计**：给情绪 eval **捎带一个可选的小 delta**——偶尔（非每轮）返回一句极短的「生活动态」：

> 「TA 把桌上的书换成了新的一本。」／「窗台多了一盆花。」／「小屋的灯还亮着。」

- 单句、极便宜，作为情绪 eval 返回体里的一个**可选字段**（`innerState`/buff 旁边加一个 `ambientEvent?`）。
- **节流双闸**（客户端判，判不过连 prompt 都不加）：时间闸（距上条 ambient < N 分钟不生成）+ 概率闸（过了时间闸也只按概率出），制造「偶尔」的惊喜感而非准点打卡。

**呈现与记忆（已拍板，两条腿）**：

1. **小屋侧——养成手游式触发点**：ambient 不是默默躺在列表里，而是在小屋画布上冒一个**引导点击的气泡/光点**（手游养成界面那种「有东西可戳」的暗示）。用户点开才展开这条动态，配合时间线沉淀成「生活动态」feed（塞 `RoomGeneratedState` 或新开 feed 结构）。
2. **聊天侧——轻量小 card，解决「角色会忘」**：每条 ambient 同时往该角色的聊天里落一张**轻量卡片**（复用 `world_card` / `phone_card` 的既有模式）。卡片进聊天 = 进上下文 = 角色下轮就知道「自己刚把书换了一本」；之后**顺着现有归档/记忆宫殿管线自然入库**，不需要为 ambient 单独发明记忆通道。

> **先例 ×2**：聊天卡片抄 `world_card`（家园演绎写回聊天的模式，`engine.ts injectWorldCard`）；worker 副作用参照 instant-push 的「副作用 directive」模式（`worker/instant-push/`）。ambient 只是这两个成熟模式的轻量组合。

## 邪修玩法扶正：小屋 = 用户拼的空间画布

**用户发明的玩法**：导入一整张家园背景图 → 导入透明 PNG 作「家具占位」→ 在背景某区域造出隐形可点击热点。

**代码现状：已完全支持**——

- 背景上传：`handleImageUpload(e,'wall'|'floor')`（`RoomApp.tsx:869`）→ base64 写入 `roomConfig.wallImage/floorImage`。
- 透明占位家具：`saveCustomItem`（`RoomApp.tsx:902`）接受任意 URL / 上传图，生成的 `RoomItem` 恒为 `isInteractive:true`（`RoomApp.tsx:845`）+ 带 `descriptionPrompt`。一张透明 PNG + 定位 `x/y/scale` = 背景图上的隐形热点。

**设计：把它扶正成一等公民，而不是隐藏的组合技。** 小屋的心智模型从「程序化家具房间」转成「**用户拼贴画布**」：

- 传一张背景（甚至就是家园场景图），丢几个透明热点。
- AI 的活儿只剩：**在这些热点上，按「当前上下文（日程当前 slot + 家园事实）」给反应**——不再有程序化家具生成。
- 这和前面三点天然合流：**没有程序化家具 → 没有每日全刷 → 只剩懒加载的热点反应 + 涓流 ambient delta。**

需要的主要是 **UI 明示这条路径**（引导用户传背景 + 放透明热点）+ 一点体验打磨，底层已通。

## 分阶段落地建议

按「风险低→高、独立可发」排序，每阶段可单独上线：

1. **阶段 A（低风险，先做）**：对齐轴 ① 家园→日程喂料。改一处 prompt 上下文，立刻让日程不再和家园相悖。（轴 ② 已白送，A 里顺手核对一下 `sim` 表现即可。）
2. **阶段 B**：小屋去每日全刷（读日程 slot + 家具懒加载缓存）。直接消灭长加载，体验提升最直观。
3. **阶段 C**：主家园唯一性（用户手动指定）。为 A/③ 提供干净的单一上游事实源与"同住人"定义。
4. **阶段 D**：对齐轴 ③ 同住人日程咬合。依赖 C 的"主家园/同住人"定义就绪后再做，否则"同住人"没有可靠出处。
5. **阶段 E**：副 API 涓流 ambient + 邪修玩法扶正。锦上添花，给小屋「活着」的感觉。

> A 和 B 可并行、互不依赖；C 让 A 更严谨（单一事实源），且是 ③ 的前置（③ 要靠"主家园同住人"圈定范围）；D 依赖 C；E 依赖 B（小屋已经不全刷了，才有地方接涓流）。

## 已拍板决策记录（设计讨论定案）

- **③ 谁当锚**：**当天谁的日程先生成谁是锚**（涌现式，不配置、不定序）。规则唯一：生成时看齐所有今天已生成的同住人。
- **ambient 呈现**：小屋里做**养成手游式的引导点击触发点**；同时落**轻量小 card 进聊天**（复用 `world_card` 模式），让角色记得自己干过啥。
- **ambient 记忆**：靠聊天卡片进上下文 → 走**现有归档/记忆宫殿管线**自然入库，不单独建记忆通道。
- **① 喂料粒度**：**都喂**（住所 + 同住人 + 时段 + 家园近期剧情摘要）——每天生成时只喂一次，成本可控。
- **`sim` 番外世界**：**坚决不注入**（不进记忆、不进日程、不进小屋），按原有 `sim` 语义，不留投射口子。
- **主家园唯一性**：用户手动指定（多 real 时弹选，不自动降级）。
- **家园触发机制**：保持现状（早/午/晚自动 tick），不改成手动。
